# Codex共同作業メモ 2026-05-14

## 共同作業の考え方

このプロトタイプは、GitHubなどのリポジトリに置くことで、Codexと人間が同じ制作物を履歴付きで共同編集できる。

Codex上での共同作業は、同時に同じ画面を編集する方式ではなく、以下の流れが基本。

1. GitHubにプロジェクトを置く
2. 修正依頼をIssue、コメント、またはCodexへの指示で残す
3. Codexがブランチを作って修正する
4. 人間がプレビューURLや差分を確認する
5. 問題なければメインへ反映する

## このプロジェクトで共有する範囲

共有対象は `CMS_prototype_20260513` フォルダだけにする。

親フォルダ全体には、制作元データ、PDF、AI、PSD、見積もり、企画書などが含まれるため、共同作業リポジトリへまとめて入れない。

## 推奨リポジトリ名

例:

- `itaya-digital-signage-cms`
- `hotel-new-itaya-signage-cms`

## ブランチ運用

- `main`: CL確認用に安定して見せる版
- `feature/...`: 新機能やデザイン修正
- `fix/...`: 表示崩れや文言修正

## Codexに頼むときの書き方

例:

```text
会場案内のライトモードだけ、1画面10件の余白を少し詰めて。
ダークモードには影響しないようにして。
修正後に http://localhost:5173/index.html?screen=venue で確認して。
```

```text
広告利用画面①の横モードで、上段と下段の画像が切り替わるタイミングを同じにして。
広告②には触らないで。
```

## CL確認用URL

公開後に以下を共有する。

- 管理画面: `公開URL/`
- 広告表示①: `公開URL/index.html?screen=ad1`
- 広告表示②: `公開URL/index.html?screen=ad2`
- 会場案内: `公開URL/index.html?screen=venue`

## 注意点

現在の試作は、登録データを各ブラウザ内に保存する。
そのため、URLを共有しても入力データは他の人と共有されない。

複数人で同じ登録データを確認したい場合は、次段階でサーバー保存型へ移行する。

## TableCheck連携時の追加方針

TableCheck APIキーや認証情報は、ブラウザ側の `app.js` には絶対に書かない。

API連携する場合は、次のように中継サーバーを用意する。

- サイネージCMS
- 中継サーバー
- TableCheck API

表示側は中継サーバーから整形済みの会場案内データだけを受け取る。
